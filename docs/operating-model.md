# Operating Model

**Initiative:** azure talos omni backbase dev
**Author:** Julio Garcia

This document describes how operators should use this repo alongside Omni to manage the `talos-omni-backbase-dev` cluster.

---

## The Two-Plane Model

This cluster operates across two management planes that own distinct concerns:

```
┌───────────────────────────────────┐   ┌───────────────────────────────────┐
│              OMNI                  │   │           THIS REPO                │
│                                   │   │                                   │
│  • Machine registration           │   │  • CNI installation (Cilium)      │
│  • Control plane bootstrap        │   │  • Helm values and configuration  │
│  • Machine config generation      │   │  • Validation and health checks   │
│  • OS upgrades                    │   │  • Runbooks and procedures        │
│  • Node scaling                   │   │  • Day-2 operations scripts       │
│  • etcd membership                │   │  • Observability access (Hubble)  │
│  • Kubernetes version upgrades    │   │                                   │
│                                   │   │                                   │
│  Access: Omni UI / API            │   │  Access: git + kubectl + helm     │
└───────────────────────────────────┘   └───────────────────────────────────┘
```

**Rule:** If the operation involves a machine, node OS, or cluster bootstrap — use Omni.
**Rule:** If the operation involves the CNI, network policy, or observability — use this repo.

## Omni Access

- **URL:** https://bellyupdown.na-west-1.omni.siderolabs.io/clusters
- **Cluster:** `talos-omni-backbase-dev`

Operations performed in Omni:
- Adding new machines (Azure VMs must be provisioned with the Omni-generated image)
- Scaling the worker count
- Scaling or replacing control plane nodes
- Upgrading Talos or Kubernetes versions
- Downloading kubeconfig
- Viewing machine health and registration status

## Operator Workflow: First-Time Setup

1. **Verify Omni cluster health** — go to Omni UI, confirm all 3 control plane nodes are healthy before proceeding
2. **Download kubeconfig** — from Omni cluster page, download kubeconfig for `talos-omni-backbase-dev`
3. **Clone this repo**
   ```bash
   git clone https://github.com/jagz70/azure-talos-omni-backbase-dev.git
   cd azure-talos-omni-backbase-dev
   ```
4. **Create `.env`**
   ```bash
   cp .env.example .env
   # Edit .env — set KUBECONFIG path and Isovalent credentials
   ```
5. **Run preflight**
   ```bash
   make preflight
   ```
6. **Install Cilium**
   ```bash
   make install
   ```
7. **Validate**
   ```bash
   make validate
   ```

## Operator Workflow: Adding a New Machine

This is an Omni-only workflow. Do NOT use `talosctl` directly.

1. Provision an Azure VM using the **Omni-generated Azure image** (not a generic Talos image)
2. The VM will boot and register with Omni automatically via the SideroLink join token
3. In Omni UI: assign the machine to `talos-omni-backbase-dev` as a worker
4. Omni will apply the appropriate machine config and join the node to the cluster
5. Cilium will automatically configure networking for the new node (no additional steps)
6. Verify node is ready: `kubectl get nodes`

**Critical:** Never provision Azure VMs with generic Talos images from the Talos GitHub releases. Only Omni-generated images contain the SideroLink configuration required for machine registration.

## What Belongs in Git

| Type | In Git | Notes |
|---|---|---|
| Helm values | Yes | `helm/isovalent-enterprise/values.yaml` and overlay |
| Deployment scripts | Yes | `scripts/` — no secrets embedded |
| Runbooks | Yes | `docs/runbooks/` |
| Architecture docs | Yes | `docs/architecture/` |
| Non-secret cluster params | Yes | `environments/dev/cluster.env` |
| kubeconfig | **No** | Download from Omni each session |
| `.env` | **No** | Operator-local, git-ignored |
| Machine configs | **No** | Omni-managed, not operator-held |
| VHD images | **No** | Binary, generated by Omni |
| Secret values | **No** | Never committed |

## Helm Release Convention

| Parameter | Value |
|---|---|
| Release name | `cilium` |
| Namespace | `kube-system` |
| Chart | `isovalent/cilium` |
| Values file | `helm/isovalent-enterprise/values.yaml` |
| Overlay | `helm/isovalent-enterprise/values.azure-talos.yaml` |

All Helm operations go through `make` targets or the `scripts/install-cilium.sh` script to ensure consistent flag usage and avoid ad hoc command history drift.
